## Q1 <a name="q1"></a>：openOii 解决了什么问题？

**标准答案：**

openOii 解决的是"AI 模型单点能力很强，但把它们接成一条可恢复、可审阅、可局部重生成的漫剧创作流程"的问题。

核心矛盾是：GPT、Stable Diffusion、Sora 等模型各自擅长一个环节，但漫剧需要十几个镜头间的角色一致性、跨阶段的审批、以及失败后不从头再来——这些需要的是**工程编排**而非**模型能力**。

论文原话（摘要）：_"传统漫画和短剧制作要经过故事整理、角色设计、分镜、画面生成、配音和审阅等环节，个人创作者很难同时处理周期、成本和一致性问题。"_

## Q2 <a name="q2"></a>：为什么选 FastAPI 而不是 Flask 或 Django？

**标准答案：**

三个决定性原因：

**① Pydantic 结构化校验（核心原因）**
openOii 的 API 涉及大量结构化字段——分镜十字段（scene/action/expression/lighting/dialogue/sfx）、WebSocket 事件模型、版本快照结构等。FastAPI 基于 Pydantic，前后端字段不一致时类型检查就能发现，比运行时报错好得多。Flask 需要手动集成 marshmallow/pydantic，Django 的 DRF 对嵌套 JSON 字段的支持不如 FastAPI 直接。

**② 原生 WebSocket + 后台任务**
生成链路是长时间任务（数分钟），HTTP 不能同步等待。FastAPI 可以用 `BackgroundTasks` / `asyncio.create_task` 启动后台生成，同一个应用内同时跑 REST API + WebSocket + 后台协程。Flask 的 WebSocket 支持靠 Flask-SocketIO 第三方库，异步支持也依赖 quart 迁移。

**③ 依赖注入**
系统用 `SessionDep`、`SettingsDep`、`WsManagerDep` 等依赖项管理数据库会话和服务实例，各 API 路由不用手动创建连接。Flask 和 Django 要手动管理请求作用域。

论文第 2.9 节：_"选型原则是实用和稳定……FastAPI 的 Pydantic 校验能在开发阶段就发现前后端字段不一致的问题。"_

## Q3 <a name="q3"></a>：工作流为什么拆成 17 个节点？为什么不是 8 个或 30 个？

**标准答案：**

17 个节点是权衡后的**中间值**，按三类划分：

| 类型 | 数量 | 节点 |
|------|------|------|
| **生产阶段** | 8 | plan_outline → plan_characters → plan_shots → render_characters → render_shots → compose_videos → compose_merge → add_audio |
| **审批门** | 6 | outline_approval / characters_approval / shots_approval / character_images_approval / shot_images_approval / compose_approval |
| **质检阶段** | 2 | critique_character_images / critique_shot_images |
| **反馈路由** | 1 | review |

**为什么不是 8 个（太粗）？**

如果只有 8 个生产节点，没有审批和质检，恢复时只能重跑整段。比如分镜图生成到一半失败了，没有任何检查点能告诉你"哪些分镜图已经生成了、哪些没生成"——只能全部重来。

**为什么不是 30 个（太细）？**

如果每个分镜的每张图都是一个独立节点，图会膨胀到不可维护。LangGraph 的状态对象也会变得非常复杂。

**openOii 的实际折中：**

- 审批和质检作为**独立节点**进入图中，因为它们涉及状态变化（approval_history、critique_scores）和副作用（WebSocket 等待事件、重生成路由）
- 生产阶段按**产物类型**拆分（大纲/角色/分镜/图像/视频），不按**个数**拆分
- 恢复时通过 `artifact_lineage` 和 `completed_stages` 判断哪些阶段已完成，跳过已完成的节点

论文第 2.6 节：_"阶段拆得越细，恢复越准确，但图会更复杂；阶段拆得太粗，实现简单，失败后却只能重跑一大段。openOii 最终拆成十七个节点，是在这两端之间取了一个中间值。"_

## Q4 <a name="q4"></a>：审批门的实现机制是什么？manual 和 auto 模式怎么切换？

**标准答案：**

审批门的实现分三个阶段：**暂停 → 等待 → 恢复**。

### 1. 暂停（_manual_approval_node, nodes.py:200）

每个审批节点（共 6 个）都调用 `_manual_approval_node`。核心逻辑：

```python
if runtime.context.auto_mode:
    return _auto_approval_result(...)   # 直接放行

# 手动模式：中断等待用户
resume_value = interrupt({"gate": gate, "message": message})
```

- `interrupt()` 是 LangGraph 的 human-in-the-loop 原语——暂停图执行，保存当前状态到 PostgreSQL Checkpointer
- 中断前会把 `run_progress` + `run_awaiting_confirm` 通过 WebSocket 推给前端
- 等待超时 1800 秒（30 分钟）

### 2. 等待（_wait_for_confirm, orchestrator.py:459）

```
前端收到 run_awaiting_confirm → 显示审批按钮 + 反馈输入框
用户点"确认"或"反馈" → WebSocket 发 {type: "confirm"}
WS handler → 保存反馈到 DB → trigger_confirm_redis(run_id)
trigger_confirm_redis → Redis SET key + PUBLISH channel
_wait_for_confirm → 订阅 Redis channel 等待信号
```

### 3. 恢复（orchestrator.py:733+）

- Redis 信号到达 → `ainvoke` 循环继续 → `Command(resume={action, feedback})` 注入图
- 审批节点根据 feedback 决定下一步：
  - **无反馈** → 进入下一生产阶段
  - **有反馈** → `review` 节点解析反馈类型，路由回大纲/角色/渲染/合成等阶段

### auto 模式切换

```
前端 ChatPanel YOLO/manual 切换按钮
  ↓
editorStore.runMode === "yolo"
  ↓
前端发 GenerateRequest: { auto_mode: true }
  ↓
_backend → runtime.context.auto_mode = True
  ↓
_manual_approval_node 直接返回 _auto_approval_result
  ↓
所有审批门自动放行，全流程 ~60 秒
```

## Q5 <a name="q5"></a>：后端恢复机制的具体流程是什么？怎么避免重复生成和重复扣费？

**标准答案：**

恢复机制回答**三个层面**：怎么触发、怎么判断恢复点、怎么避免副作用。

### 1. 触发恢复的入口

| 场景 | 触发途径 |
|------|---------|
| 页面刷新后继续 | `POST /projects/{id}/resume` |
| 新 generation 时有遗留 run | `decide_generation_entry` → `recoverable_conflict` → 前端显示恢复控制面 |
| 审批中断后恢复 | `resume_from_recovery` 从 checkpoint 恢复 |
| LangGraph 中断后自动恢复 | `_invoke_phase2_graph` 的 while 循环自动处理 |

### 2. 恢复点的判断（三重证据）

`build_recovery_summary`（run_recovery.py:207）综合三方面：

**证据 A — AgentRun 状态**
- `status`: running / awaiting_confirm / failed / cancelled
- `current_agent`: 记录上次运行到哪个 agent

**证据 B — LangGraph Checkpointer**
- 读取 PostgreSQL 中的检查点快照 → 提取 `current_stage`、`route_stage`、`stage_history`、`artifact_lineage`
- 找到 `before_stage` 对应的检查点配置，从那里继续

**证据 C — 数据库产物（"以实物为准"）**
- 从 `Stage` 表和 `Artifact` 表统计每个阶段实际产生了多少产物
- 如果 artifact_lineage 说 render_shots 完成了，但数据库里没分镜图 → 回退到渲染阶段
- **核心原则：数据库产物 > 检查点状态**

### 3. 恢复算法

```python
def _resume_target_stage(current_stage, values, completed_stages):
    route_stage = values.get("route_stage")   # review 的路由目标

    if current_stage == "review":
        return route_stage or current_stage
    if current_stage.endswith("_approval"):
        return current_stage                  # 停在审批门
    if current_stage in completed_stages:
        return _next_stage(current_stage)     # 已完成 → 跳过
    return current_stage                      # 未完成 → 继续
```

### 4. 避免重复生成的机制

| 机制 | 位置 | 说明 |
|------|------|------|
| `_should_skip_stage` | nodes.py:68 | 检查 artifact_lineage 中是否有该阶段的 key |
| `_run_sub_stage` | nodes.py:127-128 | 跳过时只返回 `{"current_stage": stage}`，不调用外部模型 |
| 幂等写入 | orchestrator | 创建角色/分镜前检查 DB 中是否已存在 |
| completed_stages | state | 记录已完成的 production stage |
| artifact_lineage | state | 记录已产生的产物 key |

```python
def _should_skip_stage(state, stage):
    artifact_lineage = state.get("artifact_lineage") or []
    return _stage_key(stage) in artifact_lineage
```

### 5. 边界情况

| 场景 | 处理方式 |
|------|---------|
| 审批途中页面关闭 | checkpoint 停在 interrupt → 恢复后重新 `_wait_for_confirm` |
| 生产阶段 crash | artifact_lineage 未记录 → 回退到当前生产阶段开头 |
| review 路由后恢复 | route_stage 记录目标阶段，直接返回 |
| video provider 未配置 | compose_videos 跳过，标记 video_generation_skipped=True |

论文第 3.8 节：_"工作流状态只负责'流程怎么走'，真正的作品内容保存在业务表里。"_

## Q6 <a name="q6"></a>：六类智能体/引擎的划分依据是什么？为什么 Review Engine 是规则引擎？

**标准答案：**

### 划分依据：按"职责边界"而非"调用模型"

| 模块 | 职责 | 调用模型 | 输出形式 |
|------|------|---------|---------|
| **Outline Agent** | 故事大纲 + 视觉圣经 | LLM | 结构化 JSON |
| **Plan Agent** | 角色设定 + 十字段分镜 | LLM | DB 实体（Character, Shot） |
| **Render Agent** | 生成角色图 + 分镜图 | LLM + Image Model | 图片 URL |
| **Critic Agent** | 图像质量 + 一致性 + 构图评分 | Image/多模态模型 | 分数 + 问题列表 |
| **Compose Agent** | 分镜视频 + 合并 + TTS/BGM | 视频/TTS 服务 | 视频/TTS 文件 URL |
| **Review Engine** | 解析反馈 → 路由到对应阶段 | **无（规则引擎）** | route_stage + rerun_mode |

### 为什么 Review Engine 不用 LLM？

三个工程理由：

**1. 确定性路由更重要**
Review Engine 只需要判断反馈中是否包含"角色"、"分镜"、"画面"、"整体"等关键词，然后路由到对应阶段。用 LLM 做这个判断引入不确定性和延迟。

**2. 恢复路径稳定可测**
规则引擎是**确定性**的——相同输入永远返回相同输出。LLM 可能今天路由到 A，明天路由到 B。

**3.** 论文第 4.3 节：_"规则能稳定解决的路由，不强行交给 LLM。"_

## Q7 <a name="q7"></a>：数据库为什么用 PostgreSQL JSON 字段存部分数据？不用 MongoDB？

**标准答案：**

### 核心原因：关系 + 半结构化的混合需求

**关系型数据**（稳定结构）：
Project → Character / Shot（一对多）、Character ↔ Shot（多对多）、Project → ArtifactVersion（版本）、Universe → SharedCharacter（跨项目）

**半结构化数据**（变长 JSON）：
- `story_outline`：三幕结构、主题、世界观——字段随模型输出变化
- `visual_bible`：视觉规则、色彩方案——格式因风格而异
- `character.reference_images`：参考图 URL 列表——数量不固定
- `report_data`：一致性报告——评分维度动态扩展

用 PostgreSQL 的 JSON/JSONB 字段存，既不需要提前定义 schema，又能和关系数据放在同一事务中。

### 对比 MongoDB

| 对比项 | PostgreSQL + JSON | MongoDB |
|--------|------------------|---------|
| 关系完整性 | 外键约束、级联删除 | 应用层维护 |
| 多表 JOIN | 原生支持 | `$lookup` 性能差 |
| 事务 | ACID 跨表和 JSON | 文档级事务 |
| 运维 | 一个数据库 | 再多维护一个 |

论文第 2.4 节：_"关系型表负责保证项目和产物之间的关系，JSON 字段负责保存变化较大的 AIGC 结果。"_

## Q8 <a name="q8"></a>：前端为什么选择 tldraw？Zustand、TanStack Query、WebSocket 各自负责什么？

**标准答案：**

### 三个状态管理层

| 层面 | 工具 | 负责内容 | 原因 |
|------|------|---------|------|
| **编辑器状态** | **Zustand** | 运行模式（yolo/manual）、画布形状、审批状态、阶段流水线 | 画布状态变化频繁，需细粒度订阅，无需服务端参与 |
| **服务端状态** | **TanStack Query** | 项目、角色、分镜、版本、Universe、一致性报告 | 缓存、防抖、后台刷新、loading/error 状态 |
| **实时事件** | **WebSocket** | run_progress、run_awaiting_confirm、critique_result 等 | 服务端主动推送，避免轮询 |

**分工原则**：Zustand 管"用户操作了什么"，TanStack Query 管"服务端有什么"，WebSocket 管"服务端刚刚发生了什么变化"。

### 为什么用 tldraw 而不是普通表单

论文第 2.3 节：_"把角色和分镜放到无限画布里……审阅漫剧时用户经常需要同时看角色、镜头顺序和最终视频，画布比列表更接近分镜工作台。"_

具体实现：
- `ShapeUtil` 定义角色卡、分镜卡、大纲区、视频区
- 后端推送 → 前端映射成画布形状
- 用户移动 → 位置防抖保存
- 每个展示区用 flex column + 显式像素高度（适配 tldraw CSS transform）

### 一次完整生成的 WebSocket 事件流

```
run_started → run_progress × N → agent_thinking →
run_progress → run_awaiting_confirm → [用户确认] →
run_progress → character_created × N → shot_created × N →
run_progress → critique_result → audio_generated →
export_completed → run_completed
```

关键设计：**WebSocket 只负责实时增量，不负责最终状态**。页面刷新后 REST 重建。

## Q9 <a name="q9"></a>：Critic Agent 怎么评分？评分低了一定会重生成吗？

**标准答案：**

### 评分维度

Critic Agent 从三个维度对图像评分（0-10）：

| 维度 | 权重 | 评估内容 |
|------|------|---------|
| **一致性 (consistency)** | 0.4 | 角色外貌、服装、发型是否与参考图一致 |
| **质量 (quality)** | 0.35 | 清晰度、光线、构图、是否变形 |
| **构图 (composition)** | 0.25 | 角色位置、背景关系、是否符合分镜描述 |

总分 = `(consistency × 0.4) + (quality × 0.35) + (composition × 0.25)`

### 评分后流程

```python
CRITIQUE_SCORE_THRESHOLD = 6.0
CRITIQUE_MAX_ROUNDS = 2
```

- **总分 ≥ 6.0 或重试轮次 ≥ 2** → 进入下一阶段
- **总分 < 6.0 且重试轮次 < 2** → 回到渲染节点重生成，轮次 +1

### 设计取舍

论文第 3.3 节：_"机器质检不是替代用户判断，而是给用户多一层检查。"_

- Critic 不可用时降级为文本审查或给默认分
- 最终审批权在**人工审批门**

### 已知局限

论文第 2.8 节：_"单人正面场景下效果不错，多人同框时准确度下降明显。"_

改进方向已在数据结构上预留：`Character.reference_images` + `face_embedding` → 可接入 IP-Adapter / InstantID。

## Q11 <a name="q11"></a>：WebSocket 断线重连怎么处理？前端怎么避免旧消息污染？

**标准答案：**

### 断线重连

```
连接失败 → 指数退避重试（1s → 2s → 4s → ... → 上限 30s）
重连后 → REST 重新读取整个项目
  GET /projects/{id}
  GET /projects/{id}/characters
  GET /projects/{id}/shots
  GET /projects/{id}/versions
  GET /projects/{id}/consistency
```

### 三道防线避免旧消息污染

**防线 1 — run_id 核对**
每个事件携带 `run_id`，editorStore 检查是否匹配当前运行，不匹配则丢弃。

**防线 2 — 覆盖式更新**
允许重复的事件做覆盖式更新而非追加。后到的事件覆盖旧事件的相同字段。

**防线 3 — Redis confirm 信号去重**
订阅前先检查 Redis key 是否存在，防止信号先到后到导致重复确认。

### 前端恢复流程

```
页面刷新 → TanStack Query 重新获取 REST 数据
  → Zustand 重建画布形状
  → WebSocket 连接建立
  → 后端推送当前 run 状态
  → 若 awaiting_confirm → 显示审批按钮
  → 若 running → 显示进度条
  → 若已完成 → 显示最终结果
```

## Q13 <a name="q13"></a>：API 为什么分成 REST 和 WebSocket 两套？

**标准答案：**

### 分界原则

论文第 3.5 节：_"一类是确定性资源接口……另一类是生成控制接口，它们不会直接返回最终作品，而是创建或更新 AgentRun，再由后台工作流继续执行。"_

**REST 负责（确定性、一次请求得到明确结果）**：
- 项目 CRUD、角色/分镜查询、版本列表、一致性报告、风格模板/Universe

**WebSocket 负责（服务端主动推送）**：
- run_progress、run_awaiting_confirm、character_created、shot_created、critique_result、audio_generated、export_completed

### 为什么不纯用 REST？

- **轮询延迟**：1 秒轮询太慢，0.1 秒请求过多
- **审批等待**：后台停在 `interrupt()`，REST 无法推送"该审批了"
- **并发推送**：多个事件同时触发时，WS 一条连接搞定

### 为什么不纯用 WebSocket？

- **可靠性**：论文第 3.5 节——_"WS 只负责把变化及时送到前端，不负责保存最终状态"_，页面刷新后 REST 重建
- **缓存友好**：TanStack Query 对 REST 缓存和过期管理，WS 做不到
- **类型安全**：FastAPI + Pydantic 对 REST 运行时校验

### 核心架构

```
HTTP 请求    → 启动/控制运行    ← 确定性、可缓存、类型安全
WebSocket   → 实时进度/产物推新  ← 不可靠、可重复、覆盖更新
数据库      → 最终状态持有者     ← 页面刷新后的唯一真相源
```

## Q14 <a name="q14"></a>：为什么选 LangGraph 而不是 Airflow 或 Prefect？LangGraph 有什么坑？

**标准答案：**

### 框架对比

| 维度 | LangGraph | Airflow | Prefect |
|------|-----------|---------|---------|
| **human-in-the-loop** | 原生 `interrupt()` | 不直接支持 | `wait_for_input` 较新 |
| **状态管理** | StateGraph 类型化状态 | XCom 传递 | Parameters |
| **检查点** | PostgreSQL Checkpointer | 需自定义 | SQLAlchemy 后端 |
| **图结构** | 有向图（支持循环/条件） | DAG（无环） | 有向图（不支持循环） |
| **恢复** | 任意中断点恢复 | 失败 task 重跑 | 失败 task 重跑 |

**决定性因素**：Airflow 和 Prefect 的 DAG 定义后不会在运行中改变结构。openOii 需要在运行中决定"审批通过后继续"还是"反馈后回到前面"——这是有**循环**和**条件分支**的图，不是纯 DAG。

### LangGraph 的坑

**① 状态类型约束严格**
`Phase2State` 是 `TypedDict`，新增字段必须声明，否则编译报错。

**② PostgreSQL Checkpointer 锁问题**
Alembic upgrade 不能在异步事件循环内跑（psycopg2 同步 + asyncpg 异步死锁）。用 `subprocess.run` 在子进程跑迁移。

**③ 副作用管理需自己写**
```python
if _should_skip_stage(state, stage):
    return {"current_stage": stage}  # 跳过，不调用外部 API
```

**④ 序列化开销**
`stage_history`、`artifact_lineage` 不断增长，每个节点后都要序列化整个状态到 PostgreSQL。

## Q16 <a name="q16"></a>：角色一致性到底怎么做的？只用提示词够吗？

**标准答案：**

### 三层次方案

- `reference_images: list[str]` — 参考图 URL
- `visual_notes: list[str]` — 中文视觉描述（"短发、红色外套"）

- 单人正面场景：效果不错

## Q20 <a name="q20"></a>：task_manager 怎么工作的？为什么多 worker 会出问题？

**标准答案：**

### TaskManager 实现

```python
class TaskManager:
    def __init__(self):
        self._tasks: Dict[int, asyncio.Task] = {}  # project_id → task

    def register(project_id, task)   # 注册任务，挤掉旧任务
    def cancel(project_id) → bool    # 取消任务
    def is_running(project_id)       # 检查是否在跑
    def remove(project_id)           # 移除记录
```

就是一个**进程本地 dict**。

### 为什么多 worker 崩溃

```
Worker A                          Worker B
┌─────────────────────┐          ┌─────────────────────┐
│ _tasks = {42: Task} │          │ _tasks = {}         │
└─────────────────────┘          └─────────────────────┘

用户 POST /projects/42/generate → LB 分到 A → task 在 A 里跑
用户 POST /projects/42/cancel   → LB 分到 B → B 查自己 dict → 没有 42 → 取消失败
```

### 硬约束

`uvicorn --workers 1`。后续如果要扩展，需把 TaskManager 状态外移到 Redis：

```python
register → Redis SET {task_id, worker_id}
cancel   → Redis PUBLISH channel:project_id:42
所有 worker 订阅 → 收到后查本地 dict → cancel()
```

## Q21 <a name="q21"></a>：config 系统怎么设计的？三层优先级？

**标准答案：**

### 优先级顺序（从高到低）

```
1. DB configitem 表         ← API 设置，重启后仍生效，覆盖一切
2. 进程环境变量             ← TEXT_PROVIDER=fake uvicorn app.main:app
3. .env 文件                ← pydantic-settings 在 get_settings() 时读取
4. 代码默认值 (default=)    ← Settings 类中的 Field(default=...)
```

### 初始化流程

```
FastAPI 启动
  ↓
get_settings() 读取 .env + 环境变量 → Settings 单例
  ↓
init_db():
  1. create_all()
  2. ensure_initialized()       ← 用 .env 值插入不存在的 ConfigItem 行
  3. ensure_postgres_checkpointer_setup()
  4. apply_settings_overrides() ← 读取 DB 所有 ConfigItem 行 → 覆盖单例
```

**关键**：`ensure_initialized` 只插入不存在的 key。首次部署时 .env 写入 DB，之后改 .env 无效——DB 旧值覆盖 .env 新值。

### 实际踩坑

某次 Docker 部署时 `DATABASE_URL=localhost:5433` 写入 configitem。改为手动启动后 .env 的 `localhost:5432` 被 DB 覆盖了，后端连不上 PostgreSQL。

**修复**：`DELETE FROM configitem WHERE key = 'DATABASE_URL'`。

### 类型转换

ConfigService.build_settings_overrides() 用 TypeAdapter 把字符串转成正确类型：
- `"true"` → `bool`
- `"18765"` → `int`
- `"0.3"` → `float`
- `["..."]` → `list`

## Q22 <a name="q22"></a>：最大工程难题及解决

**标准答案：**

### 难题一：审批中断后恢复状态不一致

**表现**：用户确认了审批却看不到新内容——系统跳过太多或重跑全部。

**根因**：只用 LangGraph 检查点做恢复，检查点说"已到 compose"但 DB 还没写完图。

**解决**：三重证据（检查点 + AgentRun + DB 产物），以数据库产物为准。

### 难题二：角色创建恢复时重复

**表现**：生成→失败→恢复→生成同名角色 "小明""小明_2""小明_3"。

**根因**：`_should_skip_stage` 只查 artifact_lineage，crash 时 lineage 还没更新。

**解决**：创建前检查 DB 是否已存在同名角色 + 幂等写入。

### 难题三：tldraw CSS transform 与 flex 冲突

**表现**：flex-wrap 不工作、高度塌陷、溢出。

**解决**：flex-wrap → CSS Grid（`min-width:0`）；百分比高度 → `position:absolute inset-0` + 像素值；`overflow:hidden` → `clip-path:inset(0)`。

### 难题四：异步 + 同步死锁

**表现**：Alembic upgrade（同步 psycopg2）阻塞 asyncpg 连接池。

**解决**：`subprocess.run(["alembic", "upgrade", "head"])` 在子进程跑。

### 难题五：task_manager 跨 worker 失效

**表现**：取消操作路由到错误 worker。

**解决**：`--workers 1` 硬约束。

### 难题六：tldraw 箭头位置错乱

**表现**：tldraw `onAfterCreate` 按 binding 触发，start binding 创建时只有 1 个 binding，箭头终端算错。

**解决**：两个 binding 都创建完后调用 `forceArrowTerminals()`。
