## 一、首页 —— 故事输入（~2 分钟）

### 🔘 打开浏览器 → `http://localhost:15173` → 展示首页

首页布局：大标题 Bangers 字体 + 副标题 + 故事文本框 + 右侧设置面板。背景有 halftone 点阵纹理。

### 📖 实现讲解

"首页承载了生成前的全部输入。核心字段绑定在 `HomePage.tsx` 的 `useState` 中：

- **故事文本**：`story` 状态，5000 字限制
- **视觉风格**：`style` 状态，来自 `STYLE_CATEGORIES` 常量——分 2D 动画/3D 风格/艺术风格三大类共 11 种

- **角色提示**：`characterHints` 动态增删，传给 PlanAgent 的 prompt
- **镜头数**：`target_shot_count`，PlanAgent prompt 中强制 LLM 生成精确数量

**IP 宇宙**（Universe）模块：`selectedUniverseId` 通过 `universesApi.list()` 查询，选择后自动计算 chapterNumber。创建项目时传入 `universe_id` 和 `chapter_number`，后续 PlanAgent 会读取 SharedCharacter 和 world_setting 作为上下文。

**生成方式**两种：`creationMode`——`"review"` 审阅模式（有审批门）或 `"quick"` 快速模式（全自动）。这个值存储到 Project 表，前端 `ProjectPage` 据此初始化 `runMode`（`"manual"` / `"yolo"`）。

点击「生成并进入画布」→ `projectsApi.create()` → `POST /api/v1/projects` → 后端 `create_project()` 写入 Project 表 → 前端 `navigate(/project/${id}?autoStart=true)`。"

## 二、画布工作台加载（~1 分钟）

### 🔘 跳转到 ProjectPage → 展示完整布局

展示：TopBar（顶部）+ StagePipeline（阶段进度）+ 中央画布（tldraw）+ 右侧 Sidebar（默认折叠）+ 右下 ChatPanel

### 📖 实现讲解

"`ProjectPage.tsx:60` 进入后干了几件事：

1. **REST 数据加载**——四个 TanStack Query 并行查询：
   - `["project", projectId]` → GET `/projects/{id}`
   - `["characters", projectId]` → GET `/projects/{id}/characters`
   - `["shots", projectId]` → GET `/projects/{id}/shots`

2. **数据喷入 Zustand**——`useEffect` 监听每个 query 完成后，通过 `editorStore.setCharacters()`、`setShots()`、`setProjectVideoUrl()` 等写入前端全局状态

3. **检测 autoStart**——`useEffect` 检测 `searchParams.get("autoStart")`，如果是 `"true"` 则触发 `generateMutation`

4. **WebSocket 连接**——`useProjectWebSocket(projectId)` 建立到 `/ws/projects/{project_id}` 的连接

状态管理分层：**Zustand** 管编辑器状态（生成状态、角色/分镜列表、画布形状），**TanStack Query** 管服务端数据缓存，**WebSocket** 管实时增量推送。

## 三、生成启动 —— 冲突检测与入口决策（~2 分钟）

### 🔘 自动触发 generate → 控制台看返回

### 📖 实现讲解

"`ProjectPage.tsx:295` `generateMutation` 调用 `projectsApi.generate(projectId, { auto_mode: store.runMode === 'yolo' })`。

后端 `generation.py:82` `generate_project()` 的核心是 `decide_generation_entry()` (`generation_entry.py`)，一个状态机决策器：

| 返回 kind | 含义 | 处理 |
|-----------|------|------|
| `active_conflict` | 有正在运行的 run | 返回 `build_recovery_control_surface()`，前端显示"恢复控制面" |
| `recoverable_conflict` | 有失败/取消的遗留 run | 同理返回恢复控制面 |
| `provider_blocked` | Provider 配置无效 | 抛出 `BusinessError(HTTP 400)` |
| `proceed` | 干净启动 | 创建 AgentRun，启动后台任务 |

这个决策发生在**创建新 run 之前**——先查 `AgentRun` 表有没有 `queued/running` 或 `failed/cancelled` 状态的遗留 run，如果有就返回 409 带控制面数据，不创建新 run。

如果 `proceed`，创建一个 `AgentRun`（status=`running`, current_agent=`orchestrator`, progress=0.0），然后 `background_tasks.add_task(_start_project_task)`。注意这个 `_start_project_task` 是异步接口层的一层包装——它负责 `asyncio.create_task` + `task_manager.register` + 异常日志。

真正干活的是 `GenerationOrchestrator`。它在后台协程里：
1. 构建 `Phase2State`
2. `build_phase2_graph()` 编译 LangGraph 状态图
3. 在 `while True` 循环里 `ainvoke` 图——这个循环处理 `interrupt()` 恢复"

## 四、LangGraph 编排 —— 17 节点工作流（~3 分钟）

### 🔘 观察画布上逐步出现：大纲 → 角色 → 分镜

### 📖 实现讲解

"整个生成链路是一张 **StateGraph**（`graph.py:9`）。共 17 个节点：

**8 个生产节点**：`plan_outline → plan_characters → plan_shots → render_characters → render_shots → compose_videos → compose_merge → add_audio`

**6 个审批门**：每个生产节点后跟一个 `*_approval` 节点

**2 个质检节点**：`critique_character_images` 和 `critique_shot_images`

**1 个反馈路由**：`review`

图结构是**线性 + 条件分支**——生产节点连到审批节点，审批节点根据用户反馈决定：无反馈→进入下一阶段，有反馈→路由到 `review`，`review` 再路由回对应的生产节点。

关键设计模式是 `_should_skip_stage()`（`nodes.py:68`）——每次进入生产节点前检查 `artifact_lineage` 里是否已有该阶段的 key。有则跳过（只返回 `{"current_stage": stage}`，不调模型），保证恢复时不重复生成。

再看 `_run_sub_stage()`（`nodes.py:89`）——每个生产节点的模板函数。它会：
1. 通过 `runtime.context.orchestrator` 获取对应 Agent 实例
2. 调用 Agent 的 `run()` 方法
3. 更新 `current_stage`、`stage_history`、`artifact_lineage`
4. 通过 WebSocket 推送 `run_progress`

**进度计算**：`workflow_progress_for_stage()`（`state.py:79`）——按 8 个生产阶段平均分配 0-1 区间，`within_stage` 参数支持阶段内子进度。

---

**第一个节点：plan_outline**

`OutlineAgent` 调用 LLM，输出结构化 JSON——包含 `logline`、三幕 `acts`（每幕有 `summary`、`key_scenes`、`emotional_arc`）、`themes`、`tone`、`visual_bible`。

生成后后端写入 Project 的 `story_outline` 字段（JSON 类型，不用预定义 schema），然后 `outline_approval` 节点触发 `_manual_approval_node`。"

## 五、审批门机制 —— Manual / Auto 模式（~2 分钟）

### 🔘 界面弹出确认按钮 → 点击确认 → 继续生成

### 📖 实现讲解

"`_manual_approval_node()` 是审批门的核心。它在 `nodes.py:200+` 附近：

```python
if runtime.context.auto_mode:
    return _auto_approval_result(...)   # 直接放行

resume_value = interrupt({"gate": gate, "message": message})
```

**手动模式**流程：
1. `interrupt()` 暂停图执行，LangGraph 保存当前状态到 **PostgreSQL Checkpointer**
2. 推送 `run_progress` 更新进度 + `run_awaiting_confirm` WebSocket 事件
3. 前端的 ChatPanel 显示审批按钮和反馈输入框
4. 用户点击确认 → WebSocket 发送 `{"type": "confirm", "data": {"run_id": ..., "feedback": ...}}`
5. WS handler 保存反馈到 DB → `trigger_confirm_redis(run_id)` → Redis `SET key` + `PUBLISH channel`
6. `_wait_for_confirm` 在后台协程里订阅 Redis channel，收到信号后继续
7. `Command(resume={"action": "confirm", "feedback": feedback})` 注入图

**自动模式**（YOLO）：
- `auto_mode=True` → `_auto_approval_result()` 直接标记 `approval_history` + 返回 `Command(resume="auto")`
- 前端 ChatPanel 有 YOLO/manual 切换按钮，状态存 `editorStore.runMode`
- 所有审批门无感通过，全流程 ~60 秒

**超时**：`_wait_for_confirm` 默认等 **1800 秒（30 分钟）**，超时后 run 标记失败。

**重连恢复**：页面刷新后，`build_recovery_summary` 通过 Checkpointer 找到 `interrupt` 位置，前端通过 `run_awaiting_confirm` 重新显示审批按钮。"

## 六、角色与分镜生成（~2 分钟）

### 🔘 画布上出现角色卡 → 分镜卡排列 → 带箭头连接

### 📖 实现讲解

**PlanAgent** 分两步：

**plan_characters**：调用 LLM 生成角色列表——每个角色含 `name`、`description`、`visual_notes`（中文视觉描述）、`image_prompt`。LLM 输出 JSON 数组，后端解析后逐个创建 Character 记录，并推 `character_created` WebSocket 事件。前端 Canvas 映射为角色卡片形状。

**plan_shots**：LLM 生成**十字段分镜**——`scene/action/expression/lighting/dialogue/sfx` 六字段 + `camera/motion_note/duration`。严格按 `target_shot_count` 输出，不多不少。每个 Shot 推 `shot_created` 事件，画布上排列为分镜卡 + 箭头。

**Prompt 工程架构**（`agents/prompts/` 下文件）：每个 Agent prompt 由三部分拼接——系统指令（角色定义 + 风格锁定规则）、用户输入（项目上下文 + 角色提示）、输出约束（JSON schema + few-shot examples）。

风格锁定映射（PlanAgent prompt 中的 "Style Locking" 节）：11 种风格各自映射到一组中文视觉描述词，例如"日系动漫→夸张表情、大眼睛、高饱和度色彩"，"国风漫画→水墨线条、低饱和度、传统服饰元素"。

## 七、图像生成 + CriticAgent 质检（~2 分钟）

### 🔘 角色图出现 → 分镜图出现 → 质检轮次

### 📖 实现讲解

**RenderAgent**（`render.py`）负责图像生成：

1. `render_characters`：为每个角色调用 Image API（`image_factory` 创建的服务——ModelScope 或 OpenAI 兼容），结合 `image_prompt` + 参考图（I2I）生成角色形象

2. 自动构建**角色圣经**（`CharacterBibleService`）——保存角色的标准图像、参考图列表、`face_embedding`（人脸向量嵌入）

3. `render_shots`：为每个分镜生成首帧图。注入三重上下文：
   - **角色身份锁**：所有在该镜头中的角色的 `visual_notes`
   - **连续性锁**：来自 `style_prompts.py` 的 `StyleLockManager`
   - **风格 prompt**：从 `style_template` 表加载

每次图像生成后，画布上的卡片图片自动更新。

**CriticAgent**（`critic.py`）评分：

| 维度 | 权重 | 评估内容 |
|------|------|---------|
| consistency | 0.4 | 与参考图/角色圣经的一致性 |
| quality | 0.35 | 清晰度、光线、变形 |
| composition | 0.25 | 角色位置、背景关系 |

总分 = `consistency×0.4 + quality×0.35 + composition×0.25`

阈值 6.0，最多重试 2 轮。`critique_round` 在 `Phase2State` 中累加，重试不足且低于阈值→路由回渲染节点重新生成。Critic 不可用时降级为文本审查或默认给分。

## 八、视频合成 + TTS/BGM 管线（~1 分钟）

### 🔘 视频区域出现 → 播放最终视频

### 📖 实现讲解

**ComposeAgent**（`compose.py`）三阶段：

1. **compose_videos**：每个分镜调视频 API（`video_factory`——OpenAI 兼容或豆包 Doubao）生成短视频片段。若 `video_provider` 未配置（`VIDEO_BASE_URL` 为空），则跳过整个 compose 阶段，标记 `video_generation_skipped=True`。

2. **compose_merge**：FFmpeg 按 `Shot.order` 排序拼接为完整视频。`image_composer.py` 处理图像序言流。

3. **add_audio**：`audio_service.py`——Edge TTS 朗读 dialogue 文本；BGM 服务匹配场景情绪（`bgm_type` 字段控制）；FFmpeg 混音叠加字幕。

**容错策略**：论文 2.5 节落地——TTS/BGM/混音任一失败时跳过该环节，保留原视频，不阻断工作流。单个分镜视频失败只重生成该分镜。

最终 `Project.video_url` 写库，`run_completed` 事件推前端。

**Webtoon 导出**：`POST /projects/{id}/export/webtoon` → `export_service.py` 遍历分镜图 → Pillow 拼接为长图 → 存入 static 目录 → `export_completed` 事件通知。"

## 九、恢复机制 —— 三重证据（~2 分钟）

### 🔘 刷新页面 → 展示恢复控制面 → 点击恢复

### 📖 实现讲解

"以下是最核心的工程实现——**可恢复性**。

**触发恢复的三个入口**：

| 场景 | 代码路径 |
|------|---------|
| 页面刷新后手动恢复 | POST `/projects/{id}/resume` → `resume_project_run()` → `orchestrator.resume_from_recovery()` |
| 新生成时有遗留 run | `decide_generation_entry()` → `recoverable_conflict` → 前端 RecoveryControl |
| 审批中断后页面恢复 | WebSocket 重连 → 后端查 run 状态 → 推 `run_awaiting_confirm` |

**恢复算法**在 `run_recovery.py:207` `build_recovery_summary()`——综合三重证据：

**证据 A — AgentRun 表**：
- `status`：`running/awaiting_confirm/failed/cancelled`
- `current_agent`：上次运行到哪个 Agent

**证据 B — LangGraph Checkpointer**（PostgreSQL）：
- 读取检查点快照，提取 `current_stage`、`route_stage`、`stage_history`、`artifact_lineage`
- 找到对应的 checkpoint 配置，从那里恢复

**证据 C — 数据库产物**：
- 查 `Stage` 表和 `Artifact` 表，统计每个阶段实际产生了多少产物
- **核心原则：DB 产物 > 检查点状态**
- 例如 lineage 说 render_shots 完成了但 DB 里分镜图不够 → 回退到渲染阶段

**恢复流程**：

```python
# orchestrator.py:733
async def resume_from_recovery(...):
    summary = await build_recovery_summary(...)
    target_stage = _resume_target_stage(...)
    # 找到 checkpoint
    config = await build_stage_recovery_config(graph, run, before_stage=target_stage)
    # 从 checkpoint 继续
    return await _invoke_phase2_graph(graph, run, config)
```

**避免重复生成的机制**：
1. `_should_skip_stage()`（nodes.py:68）——检查 artifact_lineage
2. `_run_sub_stage()`（nodes.py:127-128）——跳过时只更新 current_stage，不调用模型
3. 创建角色/分镜前查 DB 是否已存在同名实体（幂等写入）
4. `completed_stages` 和 `artifact_lineage` 记录在 Phase2State 中持久化"

## 十、反馈路由 —— ReviewRuleEngine（~1 分钟）

### 🔘 审批时输入反馈 → 观察重新生成

### 📖 实现讲解

"`ReviewRuleEngine`（`review_rules.py`）是一个**规则引擎，不是 LLM**。

`_decide_mode()` 判断反馈中是否包含"整体"、"重新"、"全部"等关键词→`"full"`（完整重生成），否则→`"incremental"`。

路由逻辑返回 `route_stage` 和 `rerun_mode`：

| 反馈关键词 | 路由目标 | 含义 |
|-----------|---------|------|
| "角色"/"人物"/"性格" | `plan_characters` | 重新规划角色 |
| "画面"/"图像"/"风格" | `render_characters` | 重新渲染 |
| "视频"/"动作"/"镜头" | `compose_videos` | 重新合成 |
| "大纲"/"故事"/"整体" | `plan_outline` | 从头来过 |

选择规则引擎而非 LLM 的三个工程理由（论文 4.3 节）：
1. **确定性**——相同输入永远返回相同输出，LLM 可能今天路由到 A 明天路由到 B
2. **低延迟**——无需调 LLM，字符串匹配即可
3. **可测试**——简单判定函数，100% 分支覆盖

反馈提交走 `POST /projects/{id}/feedback` → 创建新 AgentRun → `orchestrator.run_from_agent(agent_name="review")` 启动 review 节点。"

## 十一、取消与 task_manager 单 worker 约束（~30 秒）

### 🔘 点击「停止」→ 任务取消

### 📖 实现讲解

"`TaskManager`（`task_manager.py`）是一个进程本地 dict——`{project_id: asyncio.Task}`。

取消流程：
1. `task_manager.cancel(project_id)` → 调用 `asyncio.Task.cancel()`
2. 后台任务捕获 `CancelledError` → 标记 AgentRun.status = `cancelled`
3. 前端收到 `run_cancelled` 事件

**硬约束 `--workers 1`**：
```
Worker A: _tasks = {42: Task}
Worker B: _tasks = {}
用户 cancel → 路由到 Worker B → B 查 dict → 没有 42 → 取消失败
```
后续产品化需外移到 Redis——`task_manager.register` 改成 Redis SET `{project_id: worker_id}`，cancel 走 PUBLISH channel。

Alembic 不能在异步事件循环内跑——同步 psycopg2 与 asyncpg 连接池死锁。所以 `init_db()` 里迁移用 `subprocess.run(["alembic", "upgrade", "head"])`。"

## 十二、配置系统 —— 三层优先级（~30 秒）

### 🔘 打开设置面板 → 修改 Provider

### 📖 实现讲解

"`config.py` 的 `Settings` 类配合 `ConfigService` 实现三层优先级：

```
1. DB configitem 表     ← API 在线修改，持久化
2. 环境变量              ← 进程级覆盖
3. .env 文件             ← pydantic-settings 读取
4. 代码默认值            ← Field(default=...)
```

初始化流程（`db/session.py` `init_db()`）：
1. `create_all()` 建表
2. `ensure_initialized()` —— 用 .env 插入不存在的 ConfigItem
3. `ensure_postgres_checkpointer_setup()` —— 建 langgraph_pg_checkpointer 表
4. `apply_settings_overrides()` —— DB 配置覆盖 Settings 单例

注意：`ensure_initialized` 只插入不存在的 key。首次部署时 .env 写入 DB，之后改 .env 无效——DB 旧值覆盖 .env 新值。类型转换用 Pydantic `TypeAdapter`（`"true"` → `bool`，`"18765"` → `int`）。

Provider 工厂模式：`text_factory` / `image_factory` / `video_factory` 根据配置动态创建 TextService / ImageService / VideoService。生成启动时通过 `resolve_project_provider_settings_async()` 冻结 Provider 配置到 `agentrun.provider_snapshot`，确保恢复时使用同样的 Provider。"

## 十三、画布技术细节 — tldraw 集成（~1 分钟）

### 🔘 拖动画布上的卡片、查看右键菜单

### 📖 实现讲解

"Canvas 基于 tldraw 定制了 8 个自定义 `ShapeUtil`：大纲区、角色卡、分镜卡、视频区、文字区域等。

tldraw CSS transform 的几个坑和解决：
1. `flex-wrap` 不工作 → 改用 CSS Grid + `min-width: 0`
2. 百分比高度塌陷 → `position: absolute inset-0` + 显式像素值
3. `overflow: hidden` 失效 → `clip-path: inset(0)`
4. HTMLContainer 内子元素用 flex column + 显式 px 高度

**箭头绑定**：`onAfterCreate` 按 binding 触发——start binding 创建时只有 1 个 binding，箭头终端位置算错。修复：两个 binding 都创建完后调用 `forceArrowTerminals()`。

**useAutoFocus**（`canvas/shapes/useAutoFocus.ts`）：新形状出现时自动缩放 camera 以适配。

**updateShapes 约束**：同步更新时完全排除 `h` 字段——不覆盖 DOM 测量的实际高度值。"

## 十四、数据模型总览（~30 秒）

### 📖 快速过表结构

"15 张 SQLModel 表：核心是 `Project→Character/Shot` 一对多，`Character↔Shot` 多对多（`shot_character_binding`）。

**PostgreSQL JSON 字段**的使用（论文 2.4 节对比 MongoDB）：
- `Project.story_outline`（JSON）：三幕结构——字段随模型输出变化，不需预定义 schema
- `Project.visual_bible`：视觉规则格式因风格而异
- `Character.reference_images`：参考图 URL 数组
- `consistency_report.report_data`：评分维度动态扩展

关系型 + 半结构化的混合策略：外键约束（`project_id`）保证关系完整性，JSON 字段容纳变长 AIGC 结果。

**Universe 模块**三表：`Universe`（世界观+风格规则）→ `SharedCharacter`（可跨项目角色）→ `UniverseProjectLink`（项目关联）。角色复用流程：项目 A 创建角色→提升为 SharedCharacter→项目 B 关联同 Universe→PlanAgent 读取上下文。

**ArtifactVersion**：每次生成/反馈/手动修改/回滚触发创建新版本。回滚不删中间历史，追加新版本。"

## 十五、WebSocket 事件体系（~1 分钟）

### 🔘 浏览器 Network 面板展示 WS 消息流

### 📖 实现讲解

"`useProjectWebSocket.ts` 管理 WebSocket 连接，**27 种事件类型**。

**断线重连**：指数退避 1s→2s→4s→...→30s。重连后 REST 重新加载整个项目。

**三道防线避免旧消息污染**：
1. `run_id` 核对——editorStore 检查事件 run_id 是否匹配当前 run
2. 覆盖式更新——重复事件覆盖而非追加
3. Redis confirm 信号去重——订阅前检查 key 是否存在

**事件流示例**（一次完整生成）：
```
run_started → run_progress × N → agent_thinking →
run_progress → run_awaiting_confirm → [用户确认] →
run_progress → character_created × N → shot_created × N →
run_progress → critique_result → audio_generated →
export_completed → run_completed
```

API 分界原则（论文 3.5 节）：**REST 管确定性资源**（项目 CRUD、角色/分镜查询、版本列表），**WebSocket 管实时推送**（进度、审批、产物增量）。WS 不负责保存最终状态——页面刷新后用 REST 重建。"

## 十六、技术栈职责一览（~1 分钟）

### 📖 快速过各技术在系统中的定位

> 适合在 Q2「为什么选 FastAPI」或 Q14「为什么选 LangGraph」等追问时展开，也可作为演示收尾的总结。

---

**FastAPI**（Python web 框架）
- 提供 REST API + WebSocket 在同一进程中运行，`BackgroundTasks` 启动后台生成协程
- Pydantic 结构化校验：保证角色/分镜的十字段字段、27 种 WS 事件 schema 前后端一致
- 依赖注入：`SessionDep`、`SettingsDep`、`WsManagerDep` 管理会话和单例

**SQLModel / SQLAlchemy**（ORM）
- 15 张表映射：核心业务逻辑围绕 ORM 实体展开，schema 变更走 Alembic 迁移
- 异步会话：`AsyncSession` 配合 asyncpg，在协程中不阻塞事件循环
- JSON 字段：`story_outline`、`visual_bible` 等变长结构存 JSON，关系用外键

**PostgreSQL**
- 主数据库：存全部业务数据——项目、角色、分镜、AgentRun、消息、版本快照
- LangGraph Checkpointer：`langgraph_pg_checkpointer` 表持久化状态图检查点，支撑审批中断和恢复
- JSONB：半结构化字段无需预定义 schema

**Redis**
- 确认信号：审批门 `trigger_confirm_redis` SET key + PUBLISH channel，`_wait_for_confirm` 订阅消费
- 重连恢复：`openoii:awaiting:{run_id}` 键持久化审批门状态，刷新后可恢复

**LangGraph**（编排框架）
- 17 节点 StateGraph：计划→渲染→合成的完整生成图谱
- `interrupt()` 原语：human-in-the-loop 审批门，暂停图后持久化到 Checkpointer
- 条件边路由：`route_after_approval` 根据 feedback 决定继续/退回/review
- 检查点恢复：`aget_state_history` 查历史快照，`build_stage_recovery_config` 定位恢复点

**LangGraph Checkpointer + asyncpg**
- 每个节点执行后自动写检查点到 PostgreSQL，保障中断不丢失进度
- asyncpg 高性能异步驱动，Alembic 迁移需 `subprocess.run` 避免死锁

**Anthropic SDK / OpenAI 兼容 API**（LLM 调用）
- `LLMService`（Anthropic 原生 SDK）或 `TextService`（OpenAI 兼容接口）
- Provider 工厂模式：`text_factory` 根据 `TEXT_PROVIDER` 创建对应实例
- 生成的 LLM 输出通过 Pydantic 校验后写入业务表

**ModelScope / OpenAI Image API**（图像生成）
- `ImageService`：生成角色图和分镜图，支持 I2I 参考图
- CriticAgent 评分后，低于阈值触发重生成

**FFmpeg**（视频拼接 + 音频混音）
- `compose_merge`：`ffmpeg concat` 按 `Shot.order` 拼接短视频
- `add_audio`：将 TTS 语音 + BGM + 原视频混音输出

**Edge TTS**（语音合成）
- `audio_service.py`：读取 `Shot.dialogue` 文本，Edge TTS 朗读为音频文件

**tldraw**（画布引擎）
- 8 个自定义 `ShapeUtil` 渲染角色卡/分镜卡/大纲区/视频区
- 无限画布 + 箭头绑定 + 自定义上下文菜单
- CSS transform 定位限制：flex → Grid、百分比 → 像素值、overflow → clip-path

**React 18 + TypeScript**（前端框架）
- 函数组件 + hooks：`useEffect`、`useMutation`、`useQuery` 管理生命周期
- `tsc --noEmit` 类型检查，无 ESLint/Prettier

**Zustand**（前端状态管理）
- `editorStore`：运行模式（yolo/manual）、生成状态、角色列表、分镜列表、画布形状
- 细粒度订阅：`useEditorStore(useShallow(...))` 避免不必要的重渲染

**TanStack Query**（服务端状态缓存）
- 5 个 query key：`["project"]`、`["characters"]`、`["shots"]`、`["messages"]`、`["versions"]`
- 自动缓存 + 后台刷新 + 页面刷新后 REST 重建

**WebSocket**（实时推送）
- 27 种服务端事件：`run_progress`、`run_awaiting_confirm`、`character_created` 等
- 断线重连 + 指数退避 + run_id 核对的 three-line defense

**Tailwind CSS + DaisyUI**（样式框架）
- v3 + `@tailwind` 指令，`tailwind.config.ts` 扩展 CMYK 调色板和字体
- `@layer components` 自定义漫画风组件：`btn-doodle`、`card-comic`、`speech-bubble`

**Docker Compose**（部署编排）
- 4 个容器：postgres（5433）、redis（6380）、backend（18765）、frontend（15173）
- `uvicorn --workers 1` 硬约束
